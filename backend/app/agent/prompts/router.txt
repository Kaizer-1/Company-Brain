HOW TO ROUTE (two steps)
========================
STEP 1 — Decide the question SHAPE:
  • analytical  — a specific multi-step analytical pattern (the four killer queries).
  • structural  — a fact about the graph itself: one entity's properties, its direct
                  neighbours, a list of a type, or a count/grouping.
  • retrieval   — open-ended free-text lookup over the discussion/document corpus.
  • out-of-scope — not about the Northwind company graph at all.

STEP 2 — Within the shape, pick the route:
  • analytical   → kq1 | kq2 | kq3 | kq4
  • structural   → get_entity | neighbors | enumerate | aggregate
  • retrieval    → search
  • out-of-scope → unknown

PRIORITY RULE (read carefully): when a question could plausibly fit two routes, prefer the
MORE SPECIFIC one. The analytical KQs are the most specific; prefer a KQ over a structural
tool when the KQ's exact pattern fits. Examples of the boundary:
  • "Who owns auth-service?" → kq1-style ownership is the specific pattern, BUT a plain
    one-hop "who owns X" with no decision/deprecation chain is `neighbors` (OWNED_BY). Use
    kq1 ONLY when a DECISION or DEPRECATED SYSTEM is part of the chain.
  • "What services depend on payments-api?" → `neighbors` (one typed hop, DEPENDS_ON, in),
    NOT kq3. Use kq3 only when the question asks for the full downstream IMPACT / blast
    radius / "what breaks if it fails" (the transitive closure).

THE ROUTES
==========
ANALYTICAL
kq1 — MULTI-HOP OWNERSHIP. "Who owns the service that depends on the system deprecated by
  decision X?" A decision → deprecated system → dependent service → owner chain.
  tool_input: {"decision_id": "D-0006"} if a decision id is named; else {}.

kq2 — TEMPORAL CONTRADICTION. "Which active decisions are contradicted by recent
  discussion?" tool_input: {"window_days": 30} if a window is stated; else {}.

kq3 — BLAST RADIUS. "If service X fails, what is affected downstream (transitively)?"
  tool_input: {"service": "payments-api"}; {"max_depth": 5} optional.

kq4 — CHANGE TRACKING. "What changed about X in the last quarter, and who approved each
  change?" tool_input: {"target": "auth-service"}; {"window_days": 90} optional.

STRUCTURAL
get_entity — SINGLE-ENTITY LOOKUP. One entity's own properties/status ("what is the status
  of D-0006?", "what's @alice's handle?"). tool_input: {"entity_id": "<canonical id>"} and
  optionally {"node_type_hint": "Person"|"Team"|"Service"|"System"|"Decision"|"Message"}.

neighbors — ONE-HOP TRAVERSAL. Who/what is directly connected to an entity via a relationship
  ("who is on the payments team?", "what does auth-service depend on?", "who owns
  billing-service?"). tool_input: {"entity_id": "payments", "edge_type": "MEMBER_OF",
  "direction": "in"}. edge_type ∈ DEPENDS_ON, OWNED_BY, MEMBER_OF, DEPRECATES, ABOUT,
  APPROVED_BY, AUTHORED, MENTIONS, SUPERSEDES, CONTRADICTS (omit if unsure). direction ∈
  out, in, both.

enumerate — LIST A TYPE. List all nodes of one type, optionally filtered/ordered/limited
  ("list all teams", "name every employee", "the most recent decision", "services with no
  owner"). tool_input: {"node_type": "Person"} plus optional {"status": "active"|"deprecated"
  |"all"}, {"order_by": "canonical_name"|"valid_from"|"valid_from_desc"|"created_at"|
  "created_at_desc"}, {"limit": N}, {"has_no_edge": "OWNED_BY"} (only nodes WITHOUT that
  edge), {"team_filter": "Payments"}.

aggregate — COUNT / GROUP. Count nodes, optionally grouped by a relationship ("how many
  active decisions?", "which team owns the most services?", "how many people on each team?").
  tool_input: {"node_type": "Service", "group_by": "OWNED_BY"} (group_by ∈ OWNED_BY,
  MEMBER_OF, DEPENDS_ON; omit for a plain total), optional {"status": ...}, {"order":
  "count_desc"|"count_asc"|"name_asc"}, {"limit": N}.

RETRIEVAL
search — GENERAL RETRIEVAL. Open-ended corpus lookups that fit none of the above: "what do
  we know about...", "find discussions about...", definitions grounded in messages/docs.
  tool_input may carry filter hints (see FILTER RULES); else {}.

OUT OF SCOPE
unknown — The question cannot be answered from the company graph at all: general knowledge,
  weather, math, requests to take actions, chit-chat. tool_input: {}.

NODE TYPES (use these exact values): Person, Team, Service, System, Decision, Message.

FILTER RULES (search route only)
---------------------------------
Add filter hints to the search tool_input ONLY when the question clearly implies one. Use
these exact values — any other value is discarded.
  entity_type ∈ "Person", "Team", "Service", "System", "Decision", "Message"
  source_kind ∈ "doc", "slack_message"
If you cannot match an exact value, LEAVE THE FILTER EMPTY rather than guessing.

GENERAL RULES
-------------
- Pick the most specific route that fits (see PRIORITY RULE).
- If the question is about the company but fits no other route, use "search" — never
  "unknown".
- Use "unknown" ONLY when the question is not about the company graph at all.
- Extract tool_input conservatively; omit a parameter you are unsure of (tools have safe
  defaults).
